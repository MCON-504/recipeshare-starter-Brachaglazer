Starting Point: What the App Already Has

The RecipeShare application already has a backend structure. We have routes, templates, models, and authentication logic. Earlier, we used JSON requests for many actions because JSON is convenient for APIs, Postman, and frontend applications. A browser form is different. It gives the user a page with inputs and a button, and the browser sends the submitted values back to the server.

Today we will not try to build every form in the application at once. We will build slowly. First, we will create a form that does not touch the database. That lets us understand the mechanics without worrying about models or commits. Then we will create a form that does touch the database: a recipe form.

Focus
The goal is not to memorize syntax. The goal is to recognize the pattern. Every Flask form follows the same rhythm: create a form class, render it on GET, validate it on POST, then either show errors or perform the action.

What Is a Form in a Flask Application?

A browser form is an HTML element that collects input from a user. The browser does not know about Python classes, SQLAlchemy models, or Flask routes. It only knows that when the user clicks submit, it should send named fields to a URL using an HTTP method such as POST.

Flask receives that request. Without Flask-WTF, we could read values manually from request.form. That works, but it becomes repetitive as soon as we need validation, error messages, reusable field definitions, and CSRF protection. Flask-WTF lets us describe the form in Python and then render that form in Jinja.

LayerResponsibilityExample
HTML templateDisplays the form fields and submit button{{ form.title(...) }}
Form classDefines fields and validation rulesStringField, DataRequired()
RouteCreates the form, checks submission, and decides the responseform.validate_on_submit()
Model/databaseStores valid submitted data when neededdb.session.add(recipe)
Important Distinction
A form is not the database model. A model describes stored data. A form describes user input. They often look similar, but they are not the same object and they do not have the same job.

Flask-WTF Module

Flask-WTF is the Flask integration for WTForms. WTForms provides fields and validators. Flask-WTF adds Flask-friendly behavior such as validate_on_submit() and CSRF protection. In the last class we installed Flask-WTF forms by adding them to the requirements.txt

requirements.txt
bash
# Forms
#-----
Flask-WTF==1.2.2
WTForms==3.2.1
#-----

A Flask-WTF form needs a secret key. The secret key is used to sign CSRF tokens. In a real deployed app, this value should come from an environment variable.

app/config.py
python
import os

class Config:
    SECRET_KEY = os.getenv("SECRET_KEY", "change-me-in-production")
    SQLALCHEMY_DATABASE_URI = os.getenv("DATABASE_URL", "sqlite:///recipes.db")
    SQLALCHEMY_TRACK_MODIFICATIONS = False

The Form Flow: GET and POST

A form route usually handles two different moments. On GET, the route shows an empty form. On POST, the route receives submitted data. The route does not need two different functions. One route can handle both methods.

Most Flask form routes are built around this small loop.

generic form route pattern
python
@bp.route("/some-form", methods=["GET", "POST"])
def some_form():
    form = SomeForm()

    if form.validate_on_submit():
        # POST request + all validators passed
        # do the useful action here
        flash("Success!", "success")
        return redirect(url_for("main.some_page"))

    # GET request, or POST request with validation errors
    return render_template("some_form.html", form=form)

In-Class Form 1: A Small Feedback Form

The first form should be intentionally small and low-risk. It will collect a name, an email address, a topic, and a message. We will not save it to the database yet. On success, we will flash a message and redirect. This lets us focus only on fields, validators, route flow, and template rendering.

Step 1: Create a forms file

A common structure is to place form classes in app/forms.py for a small app, or in a blueprint-specific file such as app/main/forms.py for a larger app. For this lesson, use the location that best matches the current project structure.

app/forms.py
python
from flask_wtf import FlaskForm
from wtforms import StringField, TextAreaField, SubmitField
from wtforms.validators import DataRequired, Email, Length


class FeedbackForm(FlaskForm):
    name = StringField(
        "Name",
        validators=[DataRequired(), Length(min=2, max=80)]
    )
    email = StringField(
        "Email",
        validators=[DataRequired(), Email(), Length(max=120)]
    )
    topic = StringField(
        "Topic",
        validators=[DataRequired(), Length(max=100)]
    )
    message = TextAreaField(
        "Message",
        validators=[DataRequired(), Length(min=10, max=500)]
    )
    submit = SubmitField("Send Feedback")

Step 2: Add the route

app/routes.py
python
from flask import render_template, redirect, url_for, flash
from .forms import FeedbackForm


@main_bp.route("/feedback", methods=["GET", "POST"])
def feedback():
    form = FeedbackForm()

    if form.validate_on_submit():
        flash(f"Thanks, {form.name.data}! We received your feedback.", "success")
        return redirect(url_for("main_bp.feedback"))

    return render_template("feedback.html", form=form)

Slow Down: What Happens When Validation Fails?

Validation failure is not an exception. It is a normal part of form handling. When a user forgets a required field or enters an invalid email address, validate_on_submit() returns False. The route reaches the final render_template() line again, but now the form object contains error messages.

User actionRoute resultWhat the template can show
GET /feedbackvalidate_on_submit() is falseEmpty form
POST blank formValidators failSame form plus field errors
POST invalid emailEmail() failsError under email field
POST valid formValidators passFlash message and redirect
Focus
A failed validation should usually re-render the same form instead of redirecting. That way the form object still contains the submitted values and error messages.

app/forms.py
python
from wtforms import IntegerField
from wtforms.validators import NumberRange


class RecipeForm(FlaskForm):
    title = StringField(
        "Title",
        validators=[DataRequired(), Length(min=3, max=120)]
    )
    description = TextAreaField(
        "Description",
        validators=[DataRequired(), Length(min=10, max=300)]
    )
    instructions = TextAreaField(
        "Instructions",
        validators=[DataRequired(), Length(min=10)]
    )
    prep_time = IntegerField(
        "Prep Time in Minutes",
        validators=[DataRequired(), NumberRange(min=1, max=1440)]
    )
    submit = SubmitField("Create Recipe")
Common Mistake
A browser sends submitted values as text. IntegerField helps convert the value and report an error if the user types something that cannot be treated as an integer.

Connect the Recipe Form to the Database

This route follows the same pattern as the feedback route. The difference is that success now creates a database record. If the application uses Flask-Login, this route should also require a logged-in user and should connect the new recipe to current_user.

You already have it implemented in the routes.py

app/routes.py
python
from flask_login import login_required, current_user
from .extensions import db
from .models import Recipe
from .forms import RecipeForm


@main_bp.route("/recipes/new", methods=["GET", "POST"])
@login_required
def new_recipe():
    form = RecipeForm()

    if form.validate_on_submit():
        recipe = Recipe(
            title=form.title.data.strip(),
            description=form.description.data.strip(),
            instructions=form.instructions.data.strip(),
            prep_time=form.prep_time.data,
            author=current_user,
        )
        db.session.add(recipe)
        db.session.commit()
        flash("Recipe created successfully.", "success")
        return redirect(url_for("main_bp.get_recipe", recipe_id=recipe.id))

    return render_template("recipes/recipe_form.html", form=form, title="New Recipe")

Template Pattern: Repetition Is Normal at First

The templates above repeat the same pattern for every field: label, input, error loop. That repetition is useful while learning because it makes every part visible. Later, we can reduce repetition with Jinja macros. For today, we should keep the rendering explicit so we can see exactly how each field works.

field rendering pattern
html
<div class="form-group">
  {{ form.some_field.label }}
  {{ form.some_field(class="form-control") }}
  {% for error in form.some_field.errors %}
    <div class="field-error">{{ error }}</div>
  {% endfor %}
</div>
Optional refactor after the lesson: field macro

After the form flow is comfortable, this repeated field pattern can be moved into a macro. That is useful in larger projects, but it can hide too much while we are first learning forms.

Testing the Form Flow

We can test forms with Flask's test client. In many tests, CSRF is disabled so the test can focus on our application behavior. We are not testing Flask-WTF itself. We are testing that our route, validation rules, model creation, and redirects work together. add two tests below to your test_main_routes.py file to confirm that the feedback form shows validation errors and accepts valid submissions.

tests/test_main_routes.py
python
def test_feedback_form_shows_validation_errors(client):
    response = client.post("/feedback", data={
        "name": "",
        "email": "not-an-email",
        "topic": "",
        "message": "too short",
    })

    assert response.status_code == 200
    assert b"field-error" in response.data


def test_feedback_form_success_redirects(client):
    response = client.post("/feedback", data={
        "name": "Ari",
        "email": "ari@example.com",
        "topic": "Question",
        "message": "This message is long enough.",
    }, follow_redirects=True)

    assert response.status_code == 200
    assert b"received your feedback" in response.data
Test Setup Reminder
In the test configuration, set WTF_CSRF_ENABLED = False. Otherwise the test client must include a valid CSRF token with each POST request.

Assignment: Add a Profile Form

Create a page where a logged-in user can add or update a simple profile for the RecipeShare app.

Use a protected route at /api/profile (or your app's equivalent profile route) so users can access the form directly.

Required fields

  • display_name: required, 2–80 characters
  • bio: optional, maximum 300 characters
  • favorite_cuisine: optional, maximum 80 characters
  • years_cooking: optional integer from 0 to 100

Required behavior

  • The route must require login.
  • Add a Profile link in the authenticated navbar that points to this route.
  • A GET request should display the form.
  • A POST request should validate the form.
  • Invalid submissions should show field-level error messages.
  • Valid submissions should save the profile and flash a success message.
  • The template must use the provided skeleton below. You may adjust wording, but should not redesign the page from scratch.

Suggested model

This is the minimum structure. Use the naming conventions of the existing application if they differ.

app/models.py
python
class Profile(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    display_name = db.Column(db.String(80), nullable=False)
    bio = db.Column(db.String(300))
    favorite_cuisine = db.Column(db.String(80))
    years_cooking = db.Column(db.Integer)

    user_id = db.Column(db.Integer, db.ForeignKey("users.id"), nullable=False, unique=True)
    user = db.relationship("User", backref=db.backref("profile", uselist=False))

End-of-Lesson Checklist

By the end of this lesson, we should be able to explain and build a simple Flask form from beginning to end.

  • We can explain the difference between a browser form, a Flask route, a WTForms form class, and a SQLAlchemy model.
  • We can install Flask-WTF and configure a secret key for CSRF protection.
  • We can create fields using StringField, TextAreaField, IntegerField, and SubmitField.
  • We can attach validators such as DataRequired, Email, Length, and NumberRange.
  • We can write a route that handles both GET and POST.
  • We can use form.validate_on_submit() to separate valid submissions from empty forms and invalid submissions.
  • We can render {{ form.hidden_tag() }}, labels, fields, submit buttons, and validation errors in Jinja.
  • We can create a database object from valid form data and commit it.
  • We can protect form routes with @login_required when the action belongs to an authenticated user.
  • We can test form submissions with Flask's test client.
Big Picture
Flask forms are not a separate world from routes and models. They are the bridge between a browser user and backend application logic. Once the pattern is clear, registration forms, recipe forms, profile forms, search forms, and edit forms all become variations of the same structure.
